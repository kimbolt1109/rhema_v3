## ⚠️ Important correction to the mining brief

The brief asked to mine "EN/KO files" and "Korean-specific typography/line-breaking" handling.
**Neither exists in rhema_v2's operator-UI i18n.** `SUPPORTED_LOCALES` in `src/i18n/index.ts` is
hard-coded to exactly three entries: `en`, `ar`, `he`. There is no `ko.json`, no Korean UI locale,
and no Korean-specific typography/line-break CSS anywhere in the tracked source (`git grep -i
"korean|line-break|word-break|keep-all"` across `*.ts/*.tsx/*.md` returns no hits inside `src/`).

What Korean-related handling *does* exist in v2 is a **different subsystem** — content-language,
not operator-UI chrome:
- `src-tauri/assets/bibles/seed_ko_krv.json` — a Korean KRV (개역한글 1961) Bible seed, flagged as
  a **legal problem** (PROBLEMS.md #54: "(C) [legal] Korean KRV bundled as 'public domain' without
  verification... PD status is contested... Needs legal review before any distribution").
- `src-tauri/assets/songs/seed_korean-pd.json` — Korean public-domain worship songs, same PD-risk
  category (see `docs/SEED_COPYRIGHT_PROCESS.md`).
- Rust-side Bible reference **parsing** supports Korean book names/refs (`STATUS.md`: "regex-free
  reference parser EN+Korean incl. ranges/whole-chapter/multi-digit/요한·창").
- `docs/AI_PIPELINE.md` (~line 121): a Unicode-script **language-ID heuristic** (not a model) that
  classifies caption text as English/Korean/Chinese/Japanese/Cyrillic/Arabic to drive which
  multilingual AI model stage picks it up — this is about the *live-captioned worship service*
  content, not the app's own menus/buttons.

**Takeaway for Verger:** treat "Korean scripture/song content locale" and "operator UI chrome
locale" as two unrelated i18n concerns. v2 built real infrastructure for the former (with a legal
land-mine attached — do not carry the KRV/song PD assumption into Verger without counsel) and only
ever specced EN/AR/HE for the latter. If Verger needs a Korean **UI**, it has no v2 precedent to
draw on beyond the generic i18next mechanism below — budget it as new work, not a port.

## Namespace / key-naming convention

Single flat JSON resource tree per locale (i18next default namespace, no `ns1:key2` multi-namespace
split). Top-level keys are **UI-surface groups**, not features:

```
common       — appName, back/next/skip/finish/cancel/close/save/saving/confirm/dismiss,
               select, noMatches, optional, on/off   (cross-cutting chrome atoms)
nav          — sidebar rail item labels + a11y strings (brandAriaLabel, goToDashboard)
navGroups    — sidebar section headers (service/prep/devices/output/system)
topBar       — tier switch, live/offline pills, director toggle, PANIC button + inline confirm
statusBar    — all subsystem chips (SlideMind/CaptionCore/CameraDirector/LightSense/SpeakerID/
               Stream/Service/System/Frame), status values (OFFLINE/LOCKED/IDLE/AUTO/MANUAL/
               FADING/UNKNOWN/READY/STARTING/ERROR), safe-mode + PANIC hint
commandPalette — label, searchLabel, placeholder, escClose
wizard       — FirstRunWizard: all 10 step titles + step bodies + footer nav
settings     — language picker label + hint
```

Convention observed: **`camelCase` keys**, dot-path addressed as `t("namespace.key")`. Keys are
named for the **string's role at that call site**, not a hash/id — e.g. `topBar.panicButtonTitle`,
`statusBar.warming`, `wizard.churchNamePlaceholder`. No auto-generated keys, no key = English text
convention.

## Interpolation convention

Plain i18next `{{var}}` interpolation, no pluralization plugin/CLDR rules in use. Examples pulled
verbatim from `en.json`:
- `"liveWithCount": "Live · {{count}}"` → `t("topBar.liveWithCount", { count })`
- `"director": "Director {{state}}"`
- `"warming": "{{subsystem}} WARMING"`, `"degraded": "{{subsystem}} DEGRADED"`
- `"modelBody": "...for the {{tier}} tier..."`, `"setupModels": "Set up {{tier}} models"`

`interpolation: { escapeValue: false }` is set in `index.ts` init because React already escapes —
**do not re-enable i18next's own HTML escaping**, it would double-escape.

`$t(key)` nesting syntax is supported by i18next and explicitly preserved by the pseudo-locale
transform, but no `en.json` key was found actually using `$t()` nesting — it's future-proofed, not
exercised.

## `index.ts` — exact init shape (full logic, condensed)

```ts
export const LOCALE_STORAGE_KEY = "rhema-locale";
export const PSEUDO_LOCALE = "en-XA";
export const SUPPORTED_LOCALES: { code: string; label: string }[] = [
  { code: "en", label: "English" },
  { code: "ar", label: "العربية" },   // endonym, not "Arabic"
  { code: "he", label: "עברית" },     // endonym, not "Hebrew"
];
export { RTL_LOCALES };  // re-exported from ./direction

// pseudoRequested(): true if import.meta.env.VITE_PSEUDO_LOCALE === "1"
//                     OR URLSearchParams(location.search).get("pseudo") === "1"

i18n
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    resources: { en: {...}, /* + en-XA only when pseudo mode on */ },
    lng: pseudoOn ? PSEUDO_LOCALE : undefined,   // force pseudo when requested
    fallbackLng: "en",
    supportedLngs: pseudoOn ? [PSEUDO_LOCALE, "en", "ar", "he"] : ["en", "ar", "he"],
    nonExplicitSupportedLngs: true,
    interpolation: { escapeValue: false },
    detection: {
      order: ["localStorage", "navigator"],
      lookupLocalStorage: LOCALE_STORAGE_KEY,   // "rhema-locale"
      caches: ["localStorage"],
    },
    returnNull: false,
  });
```

Language detection order is **localStorage → navigator.language → fallback `en`** — no cookie, no
querystring, no HTML `<html lang>` detection plugin option in use (though the app *writes*
`<html lang>` itself via `useDocumentDirection`, it doesn't read from it).

## RTL handling

`RTL_LOCALES = ["ar", "he", "fa", "ur"] as const` in `direction.ts` (fa/ur are speculative —
neither has a picker entry or resource bundle; only base-subtag matching, so `ar-EG` etc. also
match via `locale.toLowerCase().split("-")[0]`).

`useDocumentDirection()` — mount once near app root (v2 calls it from `Root` in `main.tsx`):
```ts
export function useDocumentDirection(): void {
  const { i18n } = useTranslation();
  useEffect(() => {
    const apply = (lng: string) => {
      const dir = isRtlLocale(lng) ? "rtl" : "ltr";
      document.documentElement.setAttribute("dir", dir);
      document.documentElement.setAttribute("lang", lng);
    };
    apply(i18n.language || "en");
    i18n.on("languageChanged", apply);
    return () => { i18n.off("languageChanged", apply); };
  }, [i18n]);
}
```
Sets `<html dir>` + `<html lang>` directly on `document.documentElement`, reactive to i18next's
`languageChanged` event.

Styling guidance recorded in `docs/I18N.md`: prefer **CSS logical properties**
(`margin-inline`, `padding-inline`, `inset-inline`, `text-align: start`) over physical
`margin-left/right` in any touched styles; Tailwind's logical utilities (`ms-*`, `me-*`, `ps-*`,
`pe-*`, `start-*`, `end-*`) are called out as the preferred replacement for `ml-*`/`mr-*` in new or
edited code — this was a **project-wide convention change accompanying the i18n work**, not just
inside the i18n module.

## Pseudo-locale (`en-XA`) — generation approach, exact transform

Never hand-authored. Generated **at i18next init time** from `en.json` by deep-transforming the
tree, only when pseudo mode is on (never the default).

Algorithm (`pseudo.ts`, `pseudoString()`):
1. Split the string on `/(\{\{[^}]+\}\}|\$t\([^)]+\))/g` so `{{var}}` and `$t(key)` tokens are
   isolated and skipped by the transform (kept byte-for-byte for runtime interpolation to still
   work).
2. **Accent** every remaining Latin letter via a fixed `ACCENT_MAP` (a→á, b→ƀ, c→ç, ... A→Á, B→Ɓ,
   ... — full 52-entry map for a-z/A-Z, diacritics/lookalikes chosen to still be legible).
3. **Pad**: `padCount = max(1, round(nonWhitespaceLength * 0.4))`, capped at 12, using `·` repeat
   character — i.e. ~40% padding, so longer strings get proportionally more filler up to a 12-char
   ceiling.
4. **Bracket** the whole result: `` `⟦${accented} ${pad}⟧` ``.

Example from docs: `Console` → `⟦Çóñšóļé ····⟧`

`pseudoLocalize(tree)` recursively walks a JSON tree (object/array/string), applying
`pseudoString` to every string leaf, leaving numbers/booleans/structure untouched.

**Why this design (rationale worth keeping)**:
- Bracket pair → any visible text **without** `⟦…⟧` is provably un-extracted/hard-coded English —
  this is the entire manual QA method (see Gap section below).
- Accenting proves the string actually passed through `t()` (vs. a lookalike hard-coded string).
- Padding surfaces clipping/truncation/overflow in fixed-width UI before real translations (which
  are often 20-40% longer than English) ship.
- Interpolation/nesting tokens preserved verbatim so a pseudo-localized screen still renders live
  values correctly.

### How it's enabled
- **Build/CI:** `VITE_PSEUDO_LOCALE=1 npm run build` (or `vite`) — forces `lng: PSEUDO_LOCALE` at
  init so it needs zero user action (useful for automated screenshot/e2e diffing).
- **Ad-hoc/manual:** append `?pseudo=1` to the app URL — read via
  `URLSearchParams(window.location.search).get("pseudo")`.

### ⚠️ Gap: no CI wiring exists despite the doc's claim
Blueprint 13.3 says "Pseudo-localization run in CI catches concatenated/clipped strings," and
`docs/I18N.md` documents the `VITE_PSEUDO_LOCALE=1` build flag as the CI entry point — but
`.github/workflows/ci.yml` (read in full) has **no step that sets `VITE_PSEUDO_LOCALE`**, no
pseudo-build job, no screenshot diffing, nothing. The frontend CI job is just
`npx tsc --noEmit` → `npm run test:run` → `npm run build` (plain build, English only). The actual
detection method that shipped is **fully manual**: "run a pseudo build and look for any on-screen
text without `⟦…⟧` brackets — that is un-extracted English" (`docs/I18N.md` closing line). There is
also **no eslint plugin** (no `eslint-plugin-i18next`, no `i18next/no-literal-string`) and **no
`.eslintrc`/`eslint.config.*` at all** in the repo — confirmed by `git ls-tree` returning nothing
for either pattern. So: **the "lint/check that catches un-extracted strings" the brief asked about
does not exist as an automated check in v2.** It's a documented manual QA procedure only.

## Conversion recipe (the actual step-by-step process used, verbatim from docs/I18N.md)

1. `import { useTranslation } from "react-i18next";`
2. Inside the component: `const { t } = useTranslation();`
3. Replace each literal with `t("namespace.key")`; add the key to `en.json`, grouped by surface.
4. Dynamic values → interpolation: `t("topBar.liveWithCount", { count })` with the JSON value
   containing `{{count}}`.
5. **Convert accessible names too** — `title`, `aria-label`, `placeholder`, `alt` — not just
   visible text (cross-referenced against `docs/ACCESSIBILITY.md`, not mined here).
6. For a static array carrying display strings (nav items, wizard steps): store an i18n **key** on
   the item object (`labelKey` / `titleKey`), keep the array module-level (outside the component,
   so it isn't reallocated every render), and resolve with `t(item.labelKey)` **inside** the
   component so it re-localizes on language change. Confirmed real usage:
   ```ts
   // src/components/layout/Sidebar.tsx
   const NAV_ITEMS: NavItem[] = [
     { id: "dashboard", labelKey: "nav.console", icon: LayoutGrid },
     { id: "service",   labelKey: "nav.service", icon: Radio },
     // ...
   ];
   // render: label={t(item.labelKey)}

   // src/components/wizard/FirstRunWizard.tsx
   const STEPS: Step[] = [
     { titleKey: "wizard.stepWelcome", icon: Sparkles },
     { titleKey: "wizard.stepLegal",   icon: ScrollText },
     // ...
   ];
   // render: {t(s.titleKey)}
   ```
7. If a component reads `t()` inside `useMemo`/`useCallback`, **add `t` to the dependency array**
   — it changes identity on language change, and omitting it is a stale-translation bug source.

## Store wiring (Zustand)

`src/store/useRhemaStore.ts` mirrors the active locale as plain store state so any component can
read/react without touching i18next directly:
```ts
uiLocale: string;
setUiLocale: (locale: string) => void;
// init: uiLocale: i18n.language || "en"
// setUiLocale: (locale) => { i18n.changeLanguage(locale); set({ uiLocale: locale }); }
```
`setUiLocale` funnels through `i18n.changeLanguage(locale)` first — that's what persists to
`localStorage['rhema-locale']` via the detector's `caches` config and fires `languageChanged`
(driving `useDocumentDirection`) — *then* mirrors the value into Zustand so subscribers re-render.
Consumed in `src/pages/Settings.tsx` (General tab, "Interface language" `<select>`):
```tsx
const uiLocale = useRhemaStore((s) => s.uiLocale);
const setUiLocale = useRhemaStore((s) => s.setUiLocale);
// value={SUPPORTED_LOCALES.some(l => l.code === uiLocale) ? uiLocale : "en"}
// onChange={(e) => setUiLocale(e.target.value)}
```
Note the defensive fallback: if `uiLocale` somehow isn't one of the `SUPPORTED_LOCALES` codes
(e.g. browser reports a locale with no bundle), the `<select>` displays `"en"` rather than an
invalid/blank option.

## Unit test coverage (what was actually asserted, `i18n.test.ts`)

9 tests, pure-logic only (no DOM, no i18next singleton boot):
- `pseudoString` brackets + accents + pads a plain string, and the ASCII original must NOT survive
  verbatim in the output.
- `{{interpolation}}` and `$t(nesting)` are preserved byte-for-byte through the transform.
- `pseudoLocalize` deep-transforms every string leaf in a tree, leaves non-string leaves (numbers)
  untouched, and preserves structure (nested objects/arrays).
- `isRtlLocale` correctly flags `ar`/`he`/region-qualified (`ar-EG`) as RTL; `en`/`en-XA`/`es` as
  LTR.
- `RTL_LOCALES` contains at minimum `ar` and `he` (the "real church markets" per Blueprint 13.3).

## Verger application notes

The Tauri/Rust engine is not being ported, but **this entire i18n subsystem is pure
TypeScript/React and framework-agnostic w.r.t. Tauri** — it's the single most directly portable
piece of rhema_v2's frontend. Recommendations for Verger (Electron):

1. **Port the mechanism near-verbatim.** i18next + react-i18next +
   i18next-browser-languagedetector, bundled JSON resources (Electron is also offline-first / no
   HTTP i18n backend needed), same `LOCALE_STORAGE_KEY` pattern (rename to a Verger-specific key,
   e.g. `verger-locale`), same `SUPPORTED_LOCALES` shape (`{code, label}` with **endonyms**, not
   English names, in the label — e.g. `"ar" → "العربية"` not `"Arabic"`).
2. **Do not assume Korean UI support is a port task — it's new work.** v2 never built a `ko.json`.
   If Verger's target market includes Korean-speaking operators, budget full translation +ext
   review as net-new scope, not "copy v2's Korean file" (it doesn't exist). Keep Korean **content**
   locale (Bible/song text) and Korean **UI chrome** locale as separately tracked backlog items —
   conflating them was flagged as a risk above.
3. **Fix the CI gap this time.** v2 documented CI pseudo-locale enforcement but never wired it.
   For Verger: add an actual CI job that builds with the pseudo-locale env var forced on and runs
   a text-scrape/screenshot assertion (or at minimum a grep-based check for suspicious ASCII runs
   outside `t()` calls) so "un-extracted string" regressions fail the build, not a human's manual
   pseudo-build eyeball pass. Concretely: add an `i18n:check` script to `package.json` and a CI
   step calling it, rather than relying on the `?pseudo=1` manual workflow as the only gate.
4. **Consider `eslint-plugin-i18next` (`no-literal-string` rule) from day one.** v2 had zero
   ESLint config at all, so this was never even attempted. Electron/TS gives Verger a clean slate —
   wiring a literal-string lint rule at project scaffold time is far cheaper than a 100-component
   retrofit (which is exactly the state v2 was left in: only ~6/100 components converted).
5. **Namespace-per-surface, flat-key convention worked fine at this scale** (~150 keys, 1 locale) —
   keep it for Verger's early phases, but reconsider **splitting into real i18next namespaces**
   (`common`, `nav`, `wizard`, ... as separate JSON files loaded as i18next namespaces rather than
   one flat file) once the key count grows past what v2 ever reached, to keep diffs/PRs scoped and
   avoid one giant `en.json` merge-conflict magnet.
6. **The `labelKey`/`titleKey`-on-static-array pattern is worth keeping as a house convention** —
   it's a real, working answer to "how do you localize a module-level constant array without
   re-declaring it inside every component," verified via the Sidebar/FirstRunWizard examples above.
7. **RTL**: port `RTL_LOCALES`/`isRtlLocale`/`useDocumentDirection` as-is; keep preferring CSS
   logical properties project-wide from the start (Verger's design system should default to
   logical Tailwind utilities `ms-*/me-*/ps-*/pe-*/start-*/end-*` rather than retrofitting later —
   v2 treated this as a project-wide convention shift accompanying the i18n work, not an isolated
   module change).
8. **Legal-copy exemption pattern**: keep the "some strings are legal/must-stay-verbatim" escape
   hatch (CCLI disclaimer, EULA) as an explicit, documented, inline-flagged exception — don't let
   it become an excuse to skip real UI strings (v2's `TIER_DESCRIPTIONS` "temporarily" left in
   English is worth watching for scope-creep of that exemption).
9. **Test the pure logic, not the DOM**, for the transform/RTL functions (as v2's 9 vitest cases
   do) — cheap, fast, no i18next singleton needed; port those tests directly, they need no Tauri
   dependency at all.
